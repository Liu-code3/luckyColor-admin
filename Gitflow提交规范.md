# Gitflow 提交规范

## 1. 说明

本规范用于约束本项目后续开发节奏：

- 前端项目每次新增功能或修改功能后，必须完成一次 Git 提交。
- 每完成一个小任务，就进行一次提交。
- 使用 `gitflow` 思路管理分支。
- `git commit message` 使用简洁中文，不写冗长说明。
- 因当前仓库 `docs/` 目录被忽略，需版本化的流程规范统一放在仓库根目录维护。

## 2. 分支模型

- `main`：生产稳定分支，只接收发布或热修复合并结果。
- `develop`：日常集成分支，所有功能分支从这里切出并合回。
- `feature/<模块>-<任务>`：功能开发分支，对应一个小任务或一组强相关小任务。
- `release/<版本号>`：发版分支，用于提测、回归、修正文案与版本信息。
- `hotfix/<问题>`：线上紧急修复分支，从 `main` 切出，修复后同时回合 `main` 与 `develop`。

## 3. 小任务开发流程

1. 切换到 `develop` 并同步最新代码。
2. 从 `develop` 新建功能分支，例如：`feature/user-button-permission`。
3. 只完成一个任务清单中的明确小项，例如 `B3-05 在用户管理页接入按钮权限`。
4. 本地自检至少执行：

```powershell
pnpm build
```

5. 如果改动涉及路由、菜单、权限、登录态等关键主流程，按需要额外执行：

```powershell
pnpm test:smoke
```

6. 自检通过后提交一次，提交说明使用简洁中文。
7. 推进本地任务拆解文档中的任一任务时，该任务完成后必须立即提交一次。
8. 修复开发过程中发现的问题、回归或阻塞项时，也必须提交一次，不允许只改不提。
9. 功能分支完成后合并回 `develop`。
10. 到达一个可提测阶段时，从 `develop` 切 `release/<版本号>`。
11. 线上问题从 `main` 切 `hotfix/<问题>`，修复后同时回合 `main` 与 `develop`。

## 4. 提交要求

- 前端项目无论是新增功能还是修改已有功能，都不能跳过提交。
- 一个小任务对应一个提交，避免“顺手改一堆”。
- 推进任务拆解文档中的任务时，每完成一个任务必须提交一次。
- 修复 Bug、回归、测试失败、构建失败等问题时，每完成一次修复也必须提交一次。
- 提交信息只写结果，不写过程，不堆砌技术细节。
- 提交标题建议控制在 8 到 20 个中文字符内。
- 提交前先确认没有把无关文件一并提交。
- 涉及权限、路由、租户逻辑时，提交前必须自行验证是否存在越权或跳转异常风险。

## 5. 中文提交格式

推荐格式：

```text
<模块>：<动作><结果>
```

可直接使用的示例：

```text
登录模块：修复图标告警
路由系统：恢复动态路由
权限中心：新增权限指令
权限中心：接入用户页按钮权限
权限中心：接入角色页按钮权限
权限中心：接入菜单页按钮权限
权限中心：接入部门页按钮权限
租户中心：接入租户编辑权限
缓存机制：接入页面缓存配置
异常页面：补充404兜底
```

不建议使用的提交信息：

```text
fix
update
修改代码
继续开发
提交一下
各种优化
```

## 6. 分支命名示例

```text
feature/login-icon-fix
feature/route-keepalive
feature/user-button-permission
feature/role-button-permission
feature/menu-button-permission
feature/department-button-permission
release/v1.0.0
hotfix/login-redirect
```

## 7. 完成定义

一个小任务可以提交，至少要满足以下条件：

- 功能代码已完成。
- 相关权限、路由、状态或页面适配已同步补齐。
- 本地构建通过。
- 没有明显的越权、空指针、字段遗漏问题。
- 提交说明可让人一眼看懂本次改动目的。

## 8. 每次开始开发前先看

- 本次要做的是哪个任务编号。
- 这个任务依赖哪些前置任务。
- 当前任务应该放在哪个模块目录下。
- 是否涉及权限、路由、缓存、租户隔离。
- 本次提交标题准备写什么中文内容。
